# Todo Task Tool 使用说明

使用此工具可以为当前的编码会话创建和管理一个**结构化的任务列表（Todo List）**。  
它可以帮助你：

- 跟踪任务进度
- 组织复杂任务
- 向用户展示工作的完整性和条理性
- 让用户清楚了解当前任务进展以及整体请求的进度

---

# 何时使用此工具

在以下场景中，应主动使用该工具：

### 1. 复杂的多步骤任务
当一个任务需要 **3 个或以上明确步骤或操作** 时。

### 2. 非简单任务
任务需要：

- 仔细规划
- 多个操作
- 多个文件修改
- 多阶段执行

### 3. 用户明确要求 Todo List
当用户明确要求你使用 **todo list / 任务列表** 时。

### 4. 用户提供多个任务
例如用户给出：

- 编号列表
- 逗号分隔的任务列表
- 多个功能需求

### 5. 收到新的指令后
在收到新的需求后，应立即把用户需求记录为 Todo。

同时：

- 可以根据新信息编辑 Todo 列表
- 调整任务顺序
- 添加新的任务

### 6. 完成任务后
完成一个任务后应：

- 立即标记为 `completed`
- 添加任何新的后续任务

### 7. 开始新任务时
当你开始执行某个任务时：

- 将该任务状态标记为 `in_progress`
- 同一时间 **最好只有一个任务处于 in_progress**

原则：

- 完成当前任务
- 再开始新的任务

---

# 何时不应使用此工具

在以下情况应 **避免使用该工具**：

### 1. 只有一个简单任务

例如：

- 单个命令
- 单个代码修改

### 2. 任务非常简单

如果跟踪任务 **没有组织价值**，则无需使用。

### 3. 任务可以在不到 3 个简单步骤内完成

例如：

- 修改一行代码
- 添加一个注释
- 运行一个命令

### 4. 任务只是对话或解释

例如：

- 解释概念
- 回答问题
- 提供知识说明

---

**注意**

如果只有 **一个简单任务**，不要使用 Todo 工具。  
这种情况下，直接完成任务会更高效。

# 使用 Todo List 的示例

---

## 示例 1

**用户：**

> 我想在应用设置里增加一个深色模式开关。完成后请运行测试并构建项目。

**助手：**

我将帮助你在应用设置中添加深色模式开关。  
我先创建一个 Todo 列表来跟踪实现过程。

创建 Todo：

1. 在 Settings 页面创建深色模式切换组件
2. 添加深色模式状态管理（context/store）
3. 实现深色主题 CSS 样式
4. 更新现有组件以支持主题切换
5. 运行测试和构建流程，并修复可能出现的错误

开始执行第一个任务。

### 思考

助手使用 Todo List 的原因：

1. 深色模式是一个多步骤功能
2. 涉及 UI、状态管理和样式
3. 用户明确要求执行测试和构建
4. Todo List 可以确保流程完整执行

---

## 示例 2

**用户：**

**助手：**

我先搜索代码库中所有 `getCwd` 的出现位置。

（使用搜索工具）

发现：

- 15 处引用
- 分布在 8 个文件

我将创建 Todo List 来跟踪这些修改。

创建 Todo：

- 更新 file1.py 中的 getCwd
- 更新 file2.py 中的 getCwd
- 更新 file3.py 中的 getCwd
- 更新 file4.py 中的 getCwd
- 更新 file5.py 中的 getCwd
- 更新 file6.py 中的 getCwd
- 更新 file7.py 中的 getCwd
- 更新 file8.py 中的 getCwd

### 思考

助手使用 Todo List 的原因：

1. 任务涉及多个文件
2. 步骤较多
3. Todo List 可以避免遗漏
4. 可以保证代码一致性

---

## 示例 3

**用户：**

> 我要在我的电商网站实现这些功能：用户注册、商品目录、购物车、结算流程。

**助手：**

我将帮助你实现这些功能。  
首先把所有功能添加到 Todo 列表。

创建 Todo：

1. 实现用户注册系统
2. 实现商品目录
3. 实现购物车系统
4. 实现结算流程

然后从用户注册功能开始。

### 思考

助手使用 Todo List 的原因：

1. 用户提供多个功能需求
2. 功能复杂
3. 需要分步骤执行
4. Todo List 可以跟踪整体进度

---

## 示例 4

**用户：**

> 帮我优化我的 React 应用，它渲染很慢。

**助手：**

我先分析你的代码库以找出性能瓶颈。

（分析组件结构和渲染逻辑）

发现多个性能问题，于是创建 Todo List：

1. 为 ProductList 中的计算添加 memoization
2. 为 Dashboard 的长列表实现虚拟滚动
3. 优化 Gallery 组件的图片加载
4. 修复 ShoppingCart 中的状态更新循环
5. 分析 bundle 大小并实现代码拆分

### 思考

助手使用 Todo List 的原因：

1. 性能优化是复杂任务
2. 需要多个步骤
3. 涉及多个组件
4. Todo List 可以系统化处理问题

# 任务状态管理

任务状态包括：

### pending
任务尚未开始。

### in_progress
任务正在执行。

同一时间只应有 **一个任务处于 in_progress**。

### completed
任务已成功完成。

### cancelled
任务已取消或不再需要。

---

# 任务管理规则

1. 在执行过程中 **实时更新任务状态**
2. 任务完成后 **立即标记 completed**
3. 不要一次性批量标记完成
4. 同时只有 **一个任务处于 in_progress**
5. 完成当前任务再开始新任务
6. 如果任务不再需要，应标记 `cancelled`

---

# 任务拆分原则

创建 Todo 时应：

- 任务明确
- 可执行
- 可验证
- 不要过大

复杂任务应拆分为多个小任务。

推荐：

```
1. 创建用户模型
2. 实现用户注册 API
3. 实现注册页面
4. 添加注册验证
```

不推荐：

```
1. 实现用户系统
```

---

# 总结

当任务 **复杂、多步骤或涉及多个修改点** 时，应主动使用 Todo List。

优点：

- 结构清晰
- 不易遗漏
- 方便用户理解进度
- 展示系统化思考

如果不确定是否使用 Todo List，**倾向于使用它**。
