在执行过程中需要向用户提问时使用此工具。它可以让你：

1. 收集用户偏好或需求  
2. 澄清不明确的指令  
3. 在实现过程中获取用户对方案选择的决策  
4. 向用户提供不同方向的选择  

## 使用说明

- 先自行消化上下文、代码、约束和已有信息；只有确实需要用户拍板的内容，才使用此工具提问。
- 如果已经能预见有多个独立但都需要用户决定的问题，优先一次性整理到同一个 `questions` 数组里统一提问，不要拆成多次提问。
- 同一次调用里，每个问题只承载一个决策点；多个问题按主题归并，并按决策优先级排序。
- 只有当后续问题严格依赖本轮答案、在当前还无法确定问题内容时，才允许先问一轮、拿到答案后再继续追问。
- 每个问题都支持 `custom` 字段；默认 `true`。当为 `true` 时，系统会自动在选项末尾追加“不是以上任何选项”。
- 不要手动再写“其他”“自定义输入”“都不符合”等兜底选项，避免和系统自动追加的选项重复。
- 返回的答案是一个标签数组；如果允许多选，设置 `multiple: true`。
- 如果你推荐某个选项，应将其放在列表第一位，并在标签末尾添加 `(Recommended)`。

## 提问策略

- 优先把同一任务中的范围、方案、风险取舍、输出偏好等问题一次问完，减少来回打断。
- 不要把本可同时确认的问题拆成“先问 A，等用户回答后再问 B、C”。
- 如果某些问题只是低价值细节，且不影响继续推进，应优先自行做合理假设，而不是增加问题数量。
- 如果问题之间存在明显依赖，只保留当前必须确认的前置问题，等依赖被消除后再发起下一轮提问。

## 示例

适合一次问完的情况：

```json
{
  "questions": [
    {
      "header": "实现方式",
      "question": "你希望我采用哪种实现方式？",
      "custom": true,
      "options": [
        {"label": "最小改动 (Recommended)", "description": "改动范围小，风险低"},
        {"label": "彻底重构", "description": "结构更干净，但改动更大"}
      ]
    },
    {
      "header": "测试范围",
      "question": "这次需要我补哪些测试？",
      "custom": false,
      "options": [
        {"label": "核心回归 (Recommended)", "description": "覆盖主流程和关键边界"},
        {"label": "完整补齐", "description": "补充更全面的测试用例"}
      ]
    }
  ]
}
```

可以分步提问的例外：

```text
先确认“是否保留旧接口”。
只有当用户选择“保留旧接口”后，才需要继续追问“兼容层保留多久”。
这类后续问题依赖前一题答案，当前不必提前提问。
```
