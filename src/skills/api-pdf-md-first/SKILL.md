# API PDF MD First

按下面的顺序执行，避免在长会话里重复上传和解析同一个 PDF。

## 1. 先把 PDF 变成可复用的 Markdown 资产

如果用户提供了 PDF 格式的接口文档，并且目标是基于现有代码做多轮开发：

1. 先定位 PDF 路径。
2. 检查 PDF 同目录下是否已经存在同名 `.api.md` 文件。
3. 如果 `.api.md` 已存在且内容足够完整，后续开发直接读它。
4. 如果 `.api.md` 不存在，先创建一个最小可用骨架，只写标题、来源文件、适用范围、风险备注、全局约束占位、模块目录占位。
5. 如果 `.api.md` 不存在，或明显缺失关键信息，再使用 `read_file` 读取 PDF。
6. 读取完 PDF 后，不要一次性整理完整份 `.api.md`，而是按“先骨架、后模块、再接口”的顺序多次写入。
7. 每次只补一小段稳定内容；写完后以下一版 `.api.md` 作为新的事实基线继续推进，不要在后续轮次继续反复读取 PDF。

命名规则：

- `xxx.pdf` -> `xxx.api.md`
- 文档默认与 PDF 放在同一目录

这样做的目标是：

- 降低多轮会话里的 token 消耗
- 减少重复 OCR / PDF 抽取噪声
- 让后续开发统一基于结构化接口文档推进

## 2. 增量写入的固定节奏

把 `.api.md` 的生成过程拆成多个小步骤，禁止一次性输出整份接口文档。

推荐顺序：

1. 首次使用 `write_file` 创建骨架。
2. 读取 PDF 后，先只整理全局约束，例如 Base URL、鉴权方式、通用请求头、通用响应结构。
3. 按业务模块拆分内容，每次只处理一个模块。
4. 每个模块内再按单接口补充；一个接口写完再处理下一个接口。
5. 如果单个接口仍然过长，再继续拆成“参数”“响应”“错误码”“业务约束”等更小片段补写。

默认拆分粒度固定为：

- 先模块
- 后接口
- 必要时再拆到接口内小节

不要为了减少 tool 调用次数，把多个模块或整份 PDF 一次性整理成超长输出。

## 3. 写文件时的工具约束

为了降低 tool 报错概率，写 `.api.md` 时遵守下面规则：

1. 新文件先用 `write_file` 一次写入最小骨架。
2. 已存在的 `.api.md` 在继续补充前，必须先 `read_file`。
3. 后续补充统一优先使用 `edit_file` 做局部追加或局部替换。
4. 每次编辑只改当前模块或当前接口相关的小片段，不做整文件重写。
5. 如果某次输出太长或结构不稳定，就缩小本次补充范围，而不是继续硬写完整内容。

遇到以下情况，继续缩小粒度：

- 一个模块内接口太多
- 单个接口参数表和示例过长
- 模型返回的 JSON/tool 参数开始不稳定
- `edit_file` 需要替换的大段文本过长且容易定位失败

## 4. 后续轮次默认只读 `.api.md`

在进入编码、改造、补测试、联调说明、排查字段含义等后续任务时：

- 默认优先读取 `.api.md`
- 除非 `.api.md` 缺少关键信息，否则不要重新读取 PDF
- 如果发现 `.api.md` 与用户新增要求不一致，先更新 `.api.md`，再基于更新后的内容继续开发

把 `.api.md` 视为这次长会话里的接口事实基线。

## 5. 用固定格式整理 Markdown

生成 Markdown 前，先读取：

- `references/api-markdown-template.md`
- `references/extraction-rules.md`

生成时遵守这些原则：

- 尽量保留对开发真正有价值的信息，整体风格接近原文全量
- 不做无根据推测；不明确的地方标成 `待确认`
- 优先保留接口实现、请求约束、响应结构、错误码、业务规则
- 删除页眉页脚、宣传语、重复截图转写、无意义 OCR 噪声

## 6. 推荐的多次写入示例流程

下面流程的重点是“多次小写入”，不是一次写出最终成品：

1. 先定位 `xxx.pdf`，推导出 `xxx.api.md`。
2. 如果 `xxx.api.md` 不存在，先用 `write_file` 创建骨架：
   - 文档标题
   - 来源文件
   - 转换说明
   - 风险备注
   - `## 全局约束`
   - `## 模块目录`
   - `## 接口明细`
3. 使用 `read_file` 读取 PDF，先只整理全局约束，再通过 `read_file + edit_file` 补到 `.api.md`。
4. 从 PDF 中识别出模块 A、模块 B、模块 C，先补“模块目录”和模块标题。
5. 处理模块 A 时，只写模块 A 下的接口 1。
6. 接口 1 写完后，重新以当前 `.api.md` 为基线，再写模块 A 下的接口 2。
7. 模块 A 完成后，再处理模块 B；不要把模块 A、B、C 合并为一次输出。
8. 如果接口 2 太长，就先写“方法/路径/用途/参数”，下一次再补“响应示例/错误码/业务约束”。

核心原则：

- 新文件先建骨架
- 每次只补当前最小必要片段
- 每补完一次，就以最新 `.api.md` 继续下一次
- 不为了追求一步到位而牺牲 tool 稳定性

## 7. 开发阶段如何使用这个 Markdown

当用户让你“基于接口文档继续开发”时：

1. 先读 `.api.md`
2. 提取本轮编码必需的信息，例如鉴权、路径、参数、响应结构、错误语义、状态约束
3. 再结合仓库现有分层和开发规范实现代码
4. 如果接口文档存在歧义，优先在回答中说明风险点；必要时回到 PDF 补充，不要擅自猜测

## 8. 什么时候需要回退到 PDF

仅在下面场景回退读取 PDF：

- `.api.md` 不存在
- `.api.md` 明显残缺，缺少核心接口段落
- 表格、枚举、签名算法、错误码被摘要得不够准确
- 用户明确要求你对照原始 PDF 复核

如果回退读取 PDF，完成补充后要同步更新 `.api.md`，避免后面再次重复处理。
