# API PDF MD First

按下面的顺序执行，避免在长会话里重复上传和解析同一个 PDF。

## 1. 先把 PDF 变成可复用的 Markdown 资产

如果用户提供了 PDF 格式的接口文档，并且目标是基于现有代码做多轮开发：

1. 先定位 PDF 路径。
2. 检查 PDF 同目录下是否已经存在同名 `.api.md` 文件。
3. 如果 `.api.md` 已存在且内容足够完整，后续开发直接读它。
4. 如果 `.api.md` 不存在，或明显缺失关键信息，再使用 `read_file` 读取 PDF。
5. 读取完 PDF 后，立即整理并生成 `.api.md`，不要在后续轮次继续反复读取 PDF。

命名规则：

- `xxx.pdf` -> `xxx.api.md`
- 文档默认与 PDF 放在同一目录

这样做的目标是：

- 降低多轮会话里的 token 消耗
- 减少重复 OCR / PDF 抽取噪声
- 让后续开发统一基于结构化接口文档推进

## 2. 后续轮次默认只读 `.api.md`

在进入编码、改造、补测试、联调说明、排查字段含义等后续任务时：

- 默认优先读取 `.api.md`
- 除非 `.api.md` 缺少关键信息，否则不要重新读取 PDF
- 如果发现 `.api.md` 与用户新增要求不一致，先更新 `.api.md`，再基于更新后的内容继续开发

把 `.api.md` 视为这次长会话里的接口事实基线。

## 3. 用固定格式整理 Markdown

生成 Markdown 前，先读取：

- `references/api-markdown-template.md`
- `references/extraction-rules.md`

生成时遵守这些原则：

- 尽量保留对开发真正有价值的信息，整体风格接近原文全量
- 不做无根据推测；不明确的地方标成 `待确认`
- 优先保留接口实现、请求约束、响应结构、错误码、业务规则
- 删除页眉页脚、宣传语、重复截图转写、无意义 OCR 噪声

## 4. 开发阶段如何使用这个 Markdown

当用户让你“基于接口文档继续开发”时：

1. 先读 `.api.md`
2. 提取本轮编码必需的信息，例如鉴权、路径、参数、响应结构、错误语义、状态约束
3. 再结合仓库现有分层和开发规范实现代码
4. 如果接口文档存在歧义，优先在回答中说明风险点；必要时回到 PDF 补充，不要擅自猜测

## 5. 什么时候需要回退到 PDF

仅在下面场景回退读取 PDF：

- `.api.md` 不存在
- `.api.md` 明显残缺，缺少核心接口段落
- 表格、枚举、签名算法、错误码被摘要得不够准确
- 用户明确要求你对照原始 PDF 复核

如果回退读取 PDF，完成补充后要同步更新 `.api.md`，避免后面再次重复处理。
