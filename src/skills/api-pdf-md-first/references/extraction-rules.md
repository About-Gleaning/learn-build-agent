# Extraction Rules

把 PDF 接口文档转换成 `.api.md` 时，遵守下面的提炼规则。

## 1. 提炼目标

目标不是做“全文搬运”，而是做“面向后续编码的高保真整理”。

要兼顾两点：

- 尽量接近原文全量，避免遗漏开发关键细节
- 明确删除无价值内容，避免后续轮次继续浪费 token

## 2. 优先保留什么

优先保留所有会影响实现、联调、测试、安全判断的内容：

- 接口名称、Method、Path
- Base URL、环境区分
- 鉴权方式、签名算法、时间戳规则、nonce 规则
- Header / Query / Path / Body 参数
- 字段类型、是否必填、默认值、长度限制、格式要求、枚举值
- 请求与响应示例
- 响应字段树
- 错误码、错误消息、失败条件
- 幂等性、状态流转、回调要求、分页与排序规则
- 频率限制、权限限制、租户隔离或数据范围限制

## 3. 可以压缩但不要丢失什么

下面内容可以精简表达，但不要完全丢掉：

- 背景说明
- 前置条件
- 调用顺序
- 版本差异
- 注意事项

做法：

- 用 1 到 3 行中文概括
- 如果它会直接影响实现，保留原始关键词

## 4. 应该删除什么

下面内容默认不要进入 `.api.md`：

- PDF 页码、页眉、页脚
- 公司介绍、商务宣传、营销文案
- 与接口实现无关的封面和目录重复内容
- 同一张截图在文档中重复出现后的重复转写
- 无法辨认且不影响实现的 OCR 噪声
- 只体现排版、不承载语义的装饰性文本

## 5. 如何处理不确定信息

遇到以下情况，不要自行脑补：

- 表格列错位
- 扫描件导致字段名模糊
- 示例 JSON 缺字段但参数表有定义
- 错误码说明残缺
- 鉴权、签名、加密流程表达不完整

统一处理方式：

- 在对应接口段落写 `待确认`
- 简要说明不确定原因
- 如有必要，保留 PDF 原始描述片段的短摘要，但不要长段照抄

## 6. 如何在长会话里复用

一旦 `.api.md` 生成完成，后续轮次按下面策略执行：

1. 默认只读取 `.api.md`
2. 本轮若新增从 PDF 发现的信息，先补回 `.api.md`
3. 再继续代码实现、测试补充、联调支持

不要让同一个会话在多个轮次里反复重复“读取 PDF -> 临时理解 -> 丢失上下文 -> 再读 PDF”的循环。

## 7. 面向开发的检查清单

在基于 `.api.md` 开始编码前，至少确认下面几项是否齐全：

- 是否有明确的 Method 和 Path
- 是否有鉴权与通用 Header
- 是否区分了 Query、Path、Body 参数
- 是否有成功响应结构
- 是否有错误码或失败语义
- 是否有状态流转、幂等、分页、回调等特殊规则

如果这些核心信息仍缺失，再回退到 PDF 补充。
